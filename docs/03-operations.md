# C. 運用フロー・管理システム

新サイト稼働後の日常運用と、各タスクの担当・手順を定義します。

## C-1. 役割分担

| 役割 | 担当 | 主な作業 |
|---|---|---|
| **システム保守** | 司令官 | サイト構造変更、CMS設計変更、トラブル対応、バックアップ |
| **コンテンツ運用** | KARINTOW側スタッフ | イベント追加・編集、バナー差替、モデル登録、News投稿 |
| **デザイン制作** | KARINTOW側スタッフ + Canva | バナー画像制作（テンプレート使用） |
| **現場運営** | 現場スタッフ | サイトと連動しない接客・撮影業務 |

### ⚠️ 前提

- KARINTOW側スタッフは「WordPress操作経験はあるが、ITは詳しくない」レベル
- ノーコードで完結すること
- 操作ミスでサイトが壊れない設計にすること（権限制限）

## C-2. 日常運用の典型タスク

### タスク1: 新規イベント追加

頻度: 週数回

**手順:**

1. KARINTOW担当者がCanvaでバナー画像を制作
   - 既存テンプレートから複製
   - イベント名・日付・モデル名を差し替え
   - PNG/JPGでダウンロード
2. Webflow Editor（CMSのみ編集可）にログイン
3. Events コレクションで「New Event」をクリック
4. フィールドを入力:
   - Name, Date, Category, Models（既存から選択）, Venue（既存から選択）, 料金, 説明
5. Main Banner にCanvaで作った画像をアップロード
6. Status = `予約受付中`
7. 「Save & Publish」で公開

**所要時間目安**: 1イベント10分

### タスク2: イベント情報の修正

例: 開始時刻変更、注意事項追記

**手順:**
1. Webflow Editor → Events から該当イベントを開く
2. 修正してSave & Publish

### タスク3: イベントの満員/終了処理

**手順:**
1. Webflow Editor → Events から該当イベントを開く
2. Status を `満員` または `終了` に変更
3. Save & Publish

→ 自動的にイベントカードに「満員」「終了」バッジが表示されるよう、Webflowデザイン側で条件分岐を設定する。

### タスク4: 新規モデル登録

頻度: 月数回程度

**手順:**
1. プロフィール写真をモデル本人 or 事務所から受領
2. 必要なら Canva でトリミング・加工
3. Models コレクションで「New Model」
4. 必須フィールド入力 + 写真アップロード
5. Active = ON
6. Save & Publish

### タスク5: モデル引退・非表示化

**手順:**
1. Models コレクション → 該当モデル
2. Active を OFF
3. Save & Publish

→ サイトから即座に消える（過去のイベントの出演履歴は残ったまま）

### タスク6: TOPページのヒーロー画像差替

頻度: 月1〜2回

**手順:**
1. Canva でヒーロー画像（1920×1080）制作
2. Webflow Designer（要管理者権限）でヒーロー画像を差替
3. Publish

⚠️ Webflow Designer は Editor より権限が強いため、原則司令官のみアクセス。
KARINTOW担当者には触らせない。
バナー差替頻度を上げたい場合は、Heroバナーを CMS化して Editor から扱えるようにする検討も必要。

### タスク7: お知らせ（News）投稿

**手順:**
1. Webflow Editor → News コレクション
2. New News
3. タイトル・本文・日付入力
4. Save & Publish

## C-3. 権限設計

### Webflow ユーザー権限

| ロール | 担当者 | できること |
|---|---|---|
| **Workspace Owner** | 司令官 | 全権限（プラン変更、メンバー管理、サイト削除等） |
| **Designer** | 司令官 | デザイン編集・公開・CMS編集 |
| **Editor (Content Editor)** | KARINTOW担当者 | CMS編集と限定的な静的テキスト編集のみ |

### 制約事項

- Editor は **デザイン構造を変更できない**
- Editor は CMSのスキーマ（フィールド追加削除）を変更できない
- 公開 (Publish) は Editor もできる（要設定確認）

❓ **要確認**: WebflowプランによってEditor人数や権限の細かさが変わるため、現在のプランを確認

## C-4. Canvaテンプレート運用

### テンプレート構成（予定）

司令官側で以下のテンプレートを作成し、KARINTOW共有フォルダに納品：

| テンプレート | サイズ | 用途 |
|---|---|---|
| イベントバナー（横長） | 1200×630 | イベントメインバナー |
| イベントカード正方形 | 1080×1080 | SNSシェア用 |
| ヒーロー背景 | 1920×1080 | TOP最上部 |
| カテゴリアイコン背景 | 600×600 | カテゴリカードの背景画像 |
| モデル紹介カード | 1080×1350 | SNS / モデル紹介 |
| お知らせバナー（薄め） | 1200×400 | News用 |

### 運用ルール

- ブランドカラー（赤系 + ダーク背景）を厳守
- 文字フォントはテンプレートで指定したものを使用
- ロゴは支給データから変更しない
- ❓ Canvaブランドキット機能でこれらを強制可能か検証

### Canva → Webflow の連携

- 現状: Canvaでダウンロード → Webflow Editorに手動アップロード
- 💡 将来: Canva MCP / Webflow MCP連携の自動化を検討（司令官側のメンテ作業として）

## C-5. バックアップ・復旧

### 自動バックアップ

- Webflow は **Site Backup** 機能を標準提供
  - 公開のたびに自動バックアップ作成
  - 過去30日分（プランによる）から復元可能
- CMS データの定期エクスポートは別途検討

### 手動バックアップ（推奨）

司令官が月1回実施：
- Webflow CMSコンテンツのCSVエクスポート
- 画像アセットの一括ダウンロード
- GitHubリポジトリ `karintow-gsta-legacy/backups/` にコミット

❓ Webflowの自動バックアップ機能の正確な仕様は、後でWebflow ドキュメントで確認

### 復旧手順

1. Webflow Designer → Backups
2. 復元したいバージョンを選択
3. Restore

## C-6. 障害対応・問い合わせ窓口

### 一次対応

- KARINTOW担当者がサイト不具合に気付いた場合 → 司令官にLINE/メールで連絡
- 緊急度高（サイトが完全に表示されない等）→ 即時連絡
- 緊急度低（一部表示崩れ等）→ 通常業務時間内で対応

### 司令官の対応

- Webflow ダッシュボードから状況確認
- 必要なら直前のBackupから復元
- ❓ 障害連絡フローの正式定義は要相談

## C-7. SEO・分析

### 必須設定

- Google Search Console 登録
- Google Analytics 4 (GA4) 設置
- サイトマップXML自動生成（Webflow標準）
- robots.txt 設定
- OGP / Twitter Card メタタグ設定

### 月次レポート（任意）

司令官側で月1回：
- 訪問者数推移
- 人気ページTOP10
- 流入元

❓ レポーティングが必要かKARINTOW側に確認
